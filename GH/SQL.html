시스템 기능 확장에 따른 데이터베이스 구조 변경 SQL을 누적하여 기록합니다.
V1.1 SCHEMA 2026-06-12 13:47:00

프로젝트 다차원 관리 속성 컬럼 추가

기존의 단순 사업 분류체계를 확장하여 프로젝트를 다차원적으로 관리하기 위한 4가지 핵심 속성 컬럼을 project_master 테이블에 추가합니다.

  • 연도(year): 2025년 ~ 2030년 범위 제약조건 추가
  • 반기(half): '상반기' 또는 '하반기' 값만 허용하는 제약조건 추가
  • 시나리오명(scenario): 기본안 및 시나리오 명칭 입력
  • 시나리오 설명(description): 시나리오에 대한 상세 설명 저장 (기본안일 경우 비어있음)
실행 안내: 이 스크립트는 Supabase 대시보드의 SQL Editor에서 새 쿼리를 생성하여 실행하면 즉시 반영됩니다. 기존 테이블에 데이터가 있는 상태에서도 안전하게 작동(Column 추가 및 제약조건 등록)하도록 작성되었습니다.
postgresql 
-- 1. project_master 테이블에 4가지 핵심 속성 컬럼 추가
ALTER TABLE project_master ADD COLUMN IF NOT EXISTS year INT;
ALTER TABLE project_master ADD COLUMN IF NOT EXISTS half VARCHAR(10);
ALTER TABLE project_master ADD COLUMN IF NOT EXISTS scenario VARCHAR(50);
ALTER TABLE project_master ADD COLUMN IF NOT EXISTS description TEXT;

-- 2. 연도 범위 제한 제약 조건 설정 (2025년 ~ 2030년)
ALTER TABLE project_master DROP CONSTRAINT IF EXISTS chk_project_year;
ALTER TABLE project_master ADD CONSTRAINT chk_project_year CHECK (year BETWEEN 2025 AND 2030);

-- 3. 반기 구분 제약 조건 설정 ('상반기', '하반기')
ALTER TABLE project_master DROP CONSTRAINT IF EXISTS chk_project_half;
ALTER TABLE project_master ADD CONSTRAINT chk_project_half CHECK (half IN ('상반기', '하반기'));
V1.2 SCHEMA 2026-06-12 14:10:00

master_project 마스터 테이블 재구성 및 외래키 CASCADE 적용

기존 project_master 테이블을 master_project로 대체하고, 연도/반기/시나리오/설명의 4대 속성을 테이블의 정식 스키마 컬럼으로 통합합니다. 하위 4대 실적 테이블의 외래키 제약조건도 갱신합니다.

  • 테이블명 변경: master_project로 정형화
  • CASCADE 설정: 마스터 변경 시 하위 실적 데이터가 자동 동기화(UPDATE/DELETE CASCADE)되도록 외래키 재조정
실행 안내: Supabase의 SQL Editor에서 기존 테이블의 종속성을 제거하고 새 마스터 테이블 구조를 적용하기 위해 아래 스크립트를 차례대로 실행해 주십시오.
postgresql 
-- 1. 기존 테이블 및 외래키 정리 (충돌 방지용)
DROP TABLE IF EXISTS land_costs CASCADE;
DROP TABLE IF EXISTS development_costs CASCADE;
DROP TABLE IF EXISTS supplies CASCADE;
DROP TABLE IF EXISTS recoveries CASCADE;
DROP TABLE IF EXISTS project_master CASCADE;
DROP TABLE IF EXISTS master_project CASCADE;

-- 2. 새 master_project 테이블 생성 (PK: year, half, scenario)
-- project_name과 category 컬럼은 완전히 제외되었습니다.
CREATE TABLE master_project (
    year INT NOT NULL,
    half VARCHAR(10) NOT NULL,
    scenario VARCHAR(50) NOT NULL,
    description TEXT,
    PRIMARY KEY (year, half, scenario),
    CONSTRAINT chk_project_year CHECK (year BETWEEN 2025 AND 2030),
    CONSTRAINT chk_project_half CHECK (half IN ('상반기', '하반기'))
);

-- 3. land_costs 테이블 생성 (외래키 복합 참조 설정)
CREATE TABLE land_costs (
    id SERIAL PRIMARY KEY,
    year INT NOT NULL,
    half VARCHAR(10) NOT NULL,
    scenario VARCHAR(50) NOT NULL,
    detail_item VARCHAR(200) NOT NULL,
    target_year INT NOT NULL,
    amount NUMERIC DEFAULT 0,
    FOREIGN KEY (year, half, scenario) REFERENCES master_project(year, half, scenario) ON DELETE CASCADE ON UPDATE CASCADE
);

-- 4. development_costs 테이블 생성 (외래키 복합 참조 설정)
CREATE TABLE development_costs (
    id SERIAL PRIMARY KEY,
    year INT NOT NULL,
    half VARCHAR(10) NOT NULL,
    scenario VARCHAR(50) NOT NULL,
    detail_item VARCHAR(200) NOT NULL,
    target_year INT NOT NULL,
    amount NUMERIC DEFAULT 0,
    FOREIGN KEY (year, half, scenario) REFERENCES master_project(year, half, scenario) ON DELETE CASCADE ON UPDATE CASCADE
);

-- 5. supplies 테이블 생성 (외래키 복합 참조 설정)
CREATE TABLE supplies (
    id SERIAL PRIMARY KEY,
    year INT NOT NULL,
    half VARCHAR(10) NOT NULL,
    scenario VARCHAR(50) NOT NULL,
    detail_item VARCHAR(200) NOT NULL,
    target_year INT NOT NULL,
    amount NUMERIC DEFAULT 0,
    FOREIGN KEY (year, half, scenario) REFERENCES master_project(year, half, scenario) ON DELETE CASCADE ON UPDATE CASCADE
);

-- 6. recoveries 테이블 생성 (외래키 복합 참조 설정)
CREATE TABLE recoveries (
    id SERIAL PRIMARY KEY,
    year INT NOT NULL,
    half VARCHAR(10) NOT NULL,
    scenario VARCHAR(50) NOT NULL,
    detail_item VARCHAR(200) NOT NULL,
    target_year INT NOT NULL,
    amount NUMERIC DEFAULT 0,
    FOREIGN KEY (year, half, scenario) REFERENCES master_project(year, half, scenario) ON DELETE CASCADE ON UPDATE CASCADE
);
V1.3 SECURITY 2026-06-12 14:15:00

Row Level Security (RLS) 및 로그인 사용자 권한 제한 정책 적용

데이터 보안을 강화하여 데이터베이스의 직접적인 API 호출을 통제합니다. 오직 로그인에 성공한 사용자(authenticated 역할)만 데이터를 조회, 삽입, 갱신, 삭제할 수 있도록 RLS 정책을 활성화합니다.

  • RLS 정책 활성화: 비인가된 무인증(anonymous) 요청은 모든 데이터 조회 및 변경이 차단됩니다.
  • 인증 사용자(authenticated) 권한 부여: Supabase Auth에 로그인한 세션만 모든 DML(SELECT/INSERT/UPDATE/DELETE)을 허용합니다.
보안 안내: 이 SQL을 실행하고 나면, 웹 서비스(index.html)에서 로그인하지 않은 상태에서의 Supabase 데이터 접근이 완벽히 차단됩니다. 반드시 로그인 화면을 통해 인증된 계정으로 진입해야 대시보드 및 프로젝트 관리가 활성화됩니다.
postgresql 
-- 1. 모든 테이블에 Row Level Security (RLS) 활성화
ALTER TABLE master_project ENABLE ROW LEVEL SECURITY;
ALTER TABLE land_costs ENABLE ROW LEVEL SECURITY;
ALTER TABLE development_costs ENABLE ROW LEVEL SECURITY;
ALTER TABLE supplies ENABLE ROW LEVEL SECURITY;
ALTER TABLE recoveries ENABLE ROW LEVEL SECURITY;

-- 2. master_project 테이블 RLS 정책 설정 (인증된 사용자만 접근 가능)
DROP POLICY IF EXISTS "Allow authenticated access to master_project" ON master_project;
CREATE POLICY "Allow authenticated access to master_project" ON master_project
    FOR ALL
    TO authenticated
    USING (true)
    WITH CHECK (true);

-- 3. land_costs 테이블 RLS 정책 설정 (인증된 사용자만 접근 가능)

-- 4. development_costs 테이블 RLS 정책 설정 (인증된 사용자만 접근 가능)
DROP POLICY IF EXISTS "Allow authenticated access to development_costs" ON development_costs;
